0. Phạm vi và thứ tự chạy trước khi test

Tài liệu này tổng hợp lại toàn bộ các test đúng cho 6 giai đoạn đã phân tích. Mục tiêu là giúp bạn kiểm thử hệ thống theo thứ tự rõ ràng, có dữ liệu SQL để đối chiếu, có tiêu chí đạt/không đạt và có thể dùng trực tiếp khi demo hoặc viết báo cáo.

0.1. Thứ tự dựng database khuyến nghị

1. VeTauDB_trang_thai_UPDATED_GD1_GD2.sql
2. UPDATE_DB_GD3_CLEANUP.sql
3. SEED_15_KICH_BAN_MAU_GD1_GD2_GD3.sql
4. UPDATE_DB_GD4_REFUND_ADMIN_DTO_CLEAN.sql
5. UPDATE_DB_GD5_REPORTING_CLEAN.sql
6. UPDATE_DB_GD6_SECURITY_AUDIT.sql    nếu có dùng audit log, không bắt buộc
Nếu đã chạy bản SQL cũ và thấy Query executed successfully, database thường vẫn ổn. Các bản _CLEAN dùng để tránh Error List báo trùng index khi chạy lại nhiều lần.

0.2. Thứ tự test web tổng quát

1. Clean and Build project.
2. Start server NetBeans/Tomcat.
3. Test customer flow: tìm chuyến → chọn ghế → booking-review → tạo đặt chỗ → checkout.
4. Test payment callback thành công/thất bại/chạy lại nhiều lần.
5. Test my-bookings, my-tickets, ticket-check.
6. Test return-ticket.
7. Test admin/refunds.
8. Test admin/dashboard và admin/reports.
9. Test bảo mật: chưa login, returnUrl độc hại, logout, filter.

Giai đoạn 1

Test lỗi “mất điểm nặng”

Mục tiêu Giai đoạn 1 là đảm bảo luồng tạo đặt chỗ và tạo vé không còn lỗi constraint, không phụ thuộc số tiền client gửi lên, và trạng thái ban đầu thống nhất là Chờ thanh toán.

BướcCách testKết quả đúng

2 Start server. Trang /home mở được.

BướcCách testKết quả đúng

7.3. SQL kiểm tra toàn hệ thống

-- Trạng thái đơn
SELECT trang_thai, COUNT(*) AS so_luong
FROM dbo.DAT_CHO
GROUP BY trang_thai
ORDER BY trang_thai;

-- Trạng thái vé
SELECT trang_thai, COUNT(*) AS so_luong
FROM dbo.VE
GROUP BY trang_thai
ORDER BY trang_thai;

-- Trạng thái thanh toán
SELECT trang_thai, COUNT(*) AS so_luong, COALESCE(SUM(so_tien), 0) AS tong_tien
FROM dbo.THANH_TOAN
GROUP BY trang_thai
ORDER BY trang_thai;

-- Trạng thái hoàn tiền
SELECT trang_thai, COUNT(*) AS so_luong, COALESCE(SUM(so_tien_hoan), 0) AS tong_tien
FROM dbo.HOAN_TIEN
GROUP BY trang_thai
ORDER BY trang_thai;

-- Vé trùng mã
SELECT ma_ve, COUNT(*) AS so_lan
FROM dbo.VE
GROUP BY ma_ve
HAVING COUNT(*) > 1;

-- Nhiều hoàn tiền chờ xử lý cho cùng vé
SELECT ve_id, COUNT(*) AS so_yeu_cau_cho_xu_ly
FROM dbo.HOAN_TIEN
WHERE trang_thai = N'Chờ xử lý'
GROUP BY ve_id
HAVING COUNT(*) > 1;

8. Kết luận đưa vào báo cáo

Bạn có thể dùng đoạn tổng kết sau trong báo cáo cuối kỳ:

Sau 6 giai đoạn hoàn thiện, hệ thống đã được kiểm thử theo các nhóm chức năng chính:

1. Tạo đặt chỗ và tạo vé:
   - Không còn phụ thuộc số tiền client.
   - Trạng thái ban đầu thống nhất là Chờ thanh toán.
   - Không còn lỗi CHECK constraint khi insert VE/DAT_CHO.

2. Thanh toán:
   - Callback thành công cập nhật đồng bộ THANH_TOAN, DAT_CHO, VE.
   - Callback nhiều lần không tạo vé trùng.
   - Callback thất bại không xác nhận vé sai.

3. Quản lý trạng thái:
   - Các trạng thái Chờ thanh toán, Đã thanh toán, Hết hạn, Hợp lệ, Đã trả hiển thị đúng ở customer/admin.
   - ticket-check và my-tickets phản ánh đúng trạng thái vé.

4. Trả vé và hoàn tiền:
   - Khách yêu cầu trả vé tạo HOAN_TIEN = Chờ xử lý.
   - Admin refund thống nhất theo RefundAdminDTO.
   - Duyệt/từ chối hoàn tiền cập nhật đúng HOAN_TIEN.

5. Báo cáo:
   - Dashboard và báo cáo doanh thu dùng dữ liệu thật từ database.
   - Có thống kê doanh thu, vé bán, hoàn tiền, tuyến bán chạy, phương thức thanh toán.

6. Bảo mật cơ bản:
   - URL riêng được bảo vệ bằng filter.
   - returnUrl được kiểm tra để tránh redirect ra website ngoài.
   - Session được tạo lại sau login/register.
   - Logout invalidate session.
   - Password nâng cao không triển khai trong phạm vi đề tài, được đưa vào hướng phát triển thêm.
